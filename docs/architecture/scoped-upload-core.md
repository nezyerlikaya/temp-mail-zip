# Scoped Upload Core

Uploads owns upload scopes, validation policy, temporary upload handling rules, safe path generation, filename normalization, extension and MIME validation, size and dimension limits, storage path safety, upload metadata DTOs, quarantine/staging preparation, and delivery safety rules.

It does not own avatars, media library records, email message attachment records, Blog or Knowledge Base references, public galleries, CDN integration, virus scanning, business authorization, or cleanup execution.

## Scope Registry

Initial scopes are `avatar`, `media`, `contact_attachment`, `knowledge_attachment`, `blog_media`, and `message_attachment_metadata`. Unknown scopes fail. Removed v1 systems have no scopes.

Each scope defines allowed extensions, detected MIME types, size and dimension limits, disk, scoped directory, visibility, and optional Settings-backed size key.

## File Type Policy

The foundation allows only conservative image types: `jpg`, `jpeg`, `png`, and `webp`. SVG, GIF, BMP, TIFF, HEIC, HTML, scripts, templates, executables, archives, and documents are disabled until a later owning module defines explicit policy.

## Filename And Path Rules

Original filenames are sanitized metadata only. Storage filenames are random server-generated values. Path separators, control characters, null bytes, path traversal, dangerous extensions, and dangerous double extensions are rejected.

All storage paths are relative, generated by Uploads, and remain inside the scoped directory. Raw temporary uploads must not be public. Private local storage is the default strategy.

## Validation

Validation checks file validity, non-empty size, Settings-backed byte limits, extension allow-list, server-detected MIME type, image dimensions, safe generated path, and streaming SHA-256 hash generation. Client MIME and client filename are not trusted.

## Metadata

Metadata may include sanitized original filename, generated filename, size, detected MIME type, extension, SHA-256 hash, width/height, scope, disk, and relative path. It must not include binary content, absolute paths, raw EXIF, credentials, provider payloads, or document text.

## Settings

Settings keys include byte and pixel units:

- `uploads.avatar_max_size_bytes`
- `uploads.media_max_size_bytes`
- `uploads.contact_attachment_max_size_bytes`
- `uploads.max_image_width`
- `uploads.max_image_height`
- `uploads.temp_retention_minutes`

Invalid limits fail safely. Missing Settings use safe scope defaults.

## Integration Boundaries

Avatar, Media, Contact Center, Knowledge Base, Blog, and Message Storage own their records and lifecycle. Uploads provides shared validation and safe metadata only. Public delivery requires processed safe files; private delivery requires authorization and signed URL strategy where private files are served.
